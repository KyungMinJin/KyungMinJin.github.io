### 2.1. 문제 예시 (실제 직무 상황)

**AI Vision 및 데이터 최적화 직무** 관점에서 자주 출제될 수 있는 예시 문제:

> 문제
> 
> 
> 제조 공정 중 이상 탐지 시스템을 구축하는 프로젝트가 있습니다.
> 
> 주어진 데이터는 제품의 이미지 데이터와 센서 로그(time-series)입니다.
> 
> 주어진 조건에서
> 
> 1. 이상 탐지 모델을 설계할 때 어떤 방식으로 데이터를 구성하고 어떤 알고리즘을 사용해야 하는지 제안.
> 2. 모델 성능을 개선하기 위한 지표와 실험 설계 방법을 설명하시오.
> 3. 모델을 운영 환경에 배포하는 파이프라인을 간단하게 설계하고 평가 방식을 제시하시오.

---

- 2.1 풀이 답안 템플릿

[문제 이해 및 접근 방향]

이 문제는 모델 성능 자체보다, 실제 환경에서 안정적으로 검증·운영 가능한 구조를 설계하는 게 핵심이라고 봤습니다. **제조 공정에서 발생하는 이상 상황을 이미지 및 센서 로그 기반으로 탐지하고**, 이를 **운영 환경에서 안정적으로 검증·모니터링·재학습까지 가능한 AI 시스템으로 설계**하는 문제로 이해했고, 이에 따라 데이터 분포 관리 → 모델 설계 → 평가 지표 분리 → 자동화 파이프라인까지 Global CS센터의 역할에 맞게 전체 흐름으로 접근했습니다.”

1. 데이터 구성 및 전처리 전략
    1. 데이터 특성 이해
        - 이미지 데이터:
            - 제품 외관 이미지 (정상 / 불량 / 미정)
            - 일부 불량은 localization 정보(박스/마스크) 존재
        - 로그/센서 데이터:
            - 공정 단계별 센서 값 (온도, 전류, 시간, 상태 플래그 등)
            - 이미지와 timestamp 기준으로 정렬 가능
            - 시간 순서 기반 품질·동작 로그
        - 👉 핵심: **이미지와 센서 로그를 동일한 이벤트 단위로 묶는 것**
    2. 전처리 및 품질 관리
        - **이미지**
            - 이미지 레이블링 → 불량 / 정상 분류 / Segmentation 마스크
            - **데이터 불균형**: Oversampling/augmentation
                - 조명·노이즈 변화 대응 (**augmentation**)
                    - brightness / contrast jitter
                    - gaussian noise
                - 데이터 불균형 대응
                    - 불량 클래스 **oversampling**
                    - 정상 데이터는 anomaly 학습용으로 활용
                    - focal loss 고려
            - 해상도 통일 및 crop
                - 불필요한 배경 제거 → defect 집중
        - **센서 로그 (Time-series)**
            - Timestamp 기준 정렬
            - Time-series anomaly labeling → Sliding window 기반 시퀀스 데이터 생성
                - 결측치 보간 (forward fill)
                - Sliding window (예: ±5초)
            - 이상 구간 라벨링 기준 명확화
            - **장비마다 절대값 기준이 다름 →** z-score normalization (장비 간 편차 제거)
                - 각 장비(설비)별 센서 값의 평균과 분산 차이를 제거해, 패턴 자체만 비교 가능하게 만드는 정규화 방법입니다. **평소 대비 얼마나 비정상적으로 변했는가를 학습하기 위함입니다.** 예를 들어 동일한 “정상 상태”인데도 **절대값만 보면 모델은 다른 상태로 오인**
                - 장비 A 온도 평균: 70도 / 장비 B 온도 평균: 75도
                - 📐 수식 (너무 깊게 안 가도 됨)
                    - 센서 값 (x)를 다음과 같이 변환합니다:
                    - $z = \frac{x - \mu}{\sigma}$
                    - $\mu$: 해당 장비 센서의 평균
                    - $\sigma$: 해당 장비 센서의 표준편차
                
                👉 결과적으로 평균 0, 분산 1인 분포로 변환
                
                - 장점:
                    - 특정 장비에 과적합되는 현상 방지
                    - Threshold 설정이 쉬워짐
                    - anomaly score 기준을 장비별로 따로 안 잡아도 됨
                - 🏭 제조 현장 예시로 설명하면
                    - “해당 장비 기준 평균보다 3시그마 이상 벗어났기 때문에 이상” ⭕
                - 주의할 점
                    - **장비 ID 단위로 μ, σ 계산** ⭕
                    - 전체 기간 평균 ❌ → 정상 구간 기준 평균 ⭕
                    - Drift 발생 시 재계산 필요
                        - 장비 노후화 → 분포 이동
                        - → 재학습 or 통계 갱신 트리거 필요

1. 이상 탐지 모델 설계 및 학습 전략
    1. 전체 구조
        1. **Vision 기반 이상 탐지**
            - 정상 데이터 다수 → *Reconstruction 기반 Anomaly Detection*
        2. 로그/시계열 모델 (**센서 기반 이상 탐지)**
            - 시계열 패턴 붕괴 탐지
        3. **Late Fusion**
            - 두 anomaly score를 결합해 최종 판정
        
        ---
        
    2. 모델 선택
        - 이미지 모델
            - Defect Detection: (우대사항: CNN, YOLO, SegFormer)
                - Detection: YOLO 계열
                - Segmentation 필요 시 SegFormer
                - anomaly: Backbone: ResNet 기반 Autoencoder
            - 출력: Reconstruction Error Map
            - 📌 이유
                - 불량 유형이 다양하고 라벨이 제한적일 경우, **정상 패턴을 학습하고 벗어나는 경우를 이상으로 판단**하는 방식이 효과적
                - 정상 데이터 비중이 높을 경우: **(Anomaly Detection)**
                    - Autoencoder 기반 anomaly detection 병행 고려
                    - 이미지: Autoencoder 기반 reconstruction error
        - 센서 로그 모델
            - LSTM (GPU 부족 시) 또는 Temporal Transformer
            - 입력: Sliding window 시퀀스
            - 출력: Anomaly score
        - Late fusion
            - 이미지 결과(feature) + 로그 feature를 late fusion 방식으로 결합
            - 📌 이유
                - 공정 이상은 **순간 값보다 시간 흐름의 변화**가 중요
        
        ---
        
    3. 학습 방식
        - 이미지 Autoencoder
            - Loss:
                - `L = L1 Reconstruction Loss + Structural Similarity(SSIM)`
            - 이유:
                - 픽셀 차이 + 구조적 왜곡 동시 반영
                - 미세 불량에 민감
        - 센서 모델
            - Loss:
                - MSE (예측 기반)
                - 또는 One-class objective
        - 📌 **장점**
            - 불량 라벨 부족 상황에서도 안정적 학습 가능
            - 새로운 불량 유형에도 대응 가능
2. 성능 평가 / 평가 지표 및 운영·자동화 설계
    1. 검증 전략
        - 데이터 분리
            - 시간 기준 split (Train → Validation → Test)
            - 이유:
                - 미래 데이터 일반화 성능 확인
                - 데이터 leakage 방지
    2. **평가지표**
        - 모델 성능
            - 객체 탐지(Detection)/Segmentation: mAP, IOU
            - 이상 탐지: Precision / Recall / F1
            - 시계열: AUC (anomaly score 기준)
        - 운영 관점 지표
            - 오탐률 (False Positive Rate) 미탐률 (False Negative Rate) 분리 지표
            - 공정 중단 유발률 / 불량 조기 탐지 성공률
            - 시나리오별 성공률
        - 📌 **지표 분리 이유**
            - 학습 성능 ≠ 운영 품질
            - Global CS센터 관점에서는 **오탐 비용이 매우 큼**
    3. **성능 개선**
        - Failure case 기반 데이터 재수집
        - Hard example 중심 재학습 - Hard example mining
        - Edge 환경 고려한 경량화 (Quantization)
        - Early stopping
    4. 실험 설계
        - Threshold sweep 실험
            - Threshold sweep은 모델을 평가하는 실험이 아니라,운영 기준을 설계하는 실험입니다. 이상 점수(anomaly score)의 임계값을 여러 구간으로 변화시키며, 오탐·미탐·운영 비용을 함께 고려해 최적의 기준점을 찾는 실험 방법입니다.
            - 왜 Threshold가 중요한가?
                - 이상 탐지 모델은 보통: 결과를 **0/1**로 바로 내지 않고 **연속값 점수(anomaly score)**를 출력함 👉 실제 운영에서는 **“이 점수 이상이면 이상으로 판단한다”**는 기준이 반드시 필요
                - ⚠️ Threshold를 잘못 잡으면?
                    
                    
                    | Threshold | 결과 |
                    | --- | --- |
                    | 너무 낮음 | 오탐 증가 → 불필요한 알람 / 공정 중단 |
                    | 너무 높음 | 미탐 증가 → 불량 유출 / 품질 사고 |
            
            👉 그래서 **단일 지표 최적화는 위험**
            
            ---
            
            - 🧪 Threshold Sweep 실험 방식 (단계별)
                - 1️⃣ Anomaly score 분포 분석
                    - 정상/이상 데이터에 대한 score 분포 시각화
                    - 장비별 / 공정별 분리 분석
                - 📌 목적 → 점수가 **어디서 겹치는지**, 분리 가능한지 확인
                
                ---
                
                - 2️⃣ Threshold 범위 설정
                    - score ∈ [0.0 ~ 1.0]
                    - step = 0.01 또는 percentile 기준
                
                ---
                
                - 3️⃣ Threshold별 성능 계산
                    - 각 threshold마다: Precision / Recall / F1 / False Positive Rate / False Negative Rate
                    
                    추가로 운영 관점 지표:
                    
                    - 알람 발생 빈도 / 공정 중단 횟수 / 평균 대응 시간
                
                ---
                
                - 4️⃣ Trade-off 분석
                    - Precision–Recall curve
                    - FPR vs 운영 비용
                    - 공정 영향도 기준 비교
                    
                    👉 단순 F1 최대가 아니라 **업무 목적에 맞는 최적점 선택**
                    
            
            ---
            
            - 🧠 Global CS센터 관점에서의 핵심 포인트
                - ✔ 지표 분리 사고
                    - 모델 성능 지표 ≠ 운영 품질 지표
                - 예시: 학습 기준: F1
                - 예시: 운영 기준: 오탐률 ≤ X%
            
            ---
            
            - ✔ 시나리오 기반 threshold
                - 초기 양산 단계: 보수적 (오탐 허용)
                - 안정화 단계: 공격적 (미탐 최소화)
            
            ---
            
            - 🏭 제조 현장 예시
                
                > “Threshold 0.6에서는 F1이 가장 높지만, 오탐으로 인한 공정 중단이 잦았습니다. 따라서 F1은 다소 낮지만 오탐률이 안정적인 0.72를 운영 기준 threshold로 선택했습니다.”
                > 
            
            ---
            
            - 🎤 면접에서 쓰기 좋은 답변 문장
            
            > “Threshold sweep 실험을 통해 anomaly score 기준을 단일 값으로 고정하지 않고, 오탐·미탐·운영 비용 간의 trade-off를 분석했습니다. 이를 통해 모델 성능이 아니라 **실제 공정 안정성에 맞는 threshold를 설정**할 수 있었습니다.”
            > 
            
            ---
            
            - ⚠️ 실무에서 주의할 점 (이거 말하면 플러스)
                - 1️⃣ 장비별 threshold 분리 가능성
                    - 동일 기준이 항상 최선은 아님
                - 2️⃣ 분포 drift 발생 시 재조정 필요
                    - 신규 공정 / 장비 노후화
                - 3️⃣ 자동 threshold 튜닝 트리거
                    - 오탐률 급증 시 재스윕
            
            ---
            
        - 공정별 / 설비별 성능 분리 분석
        - Failure case 유형화 후 재학습
        - 📌 **장점**
            - 운영 기준에 맞는 최적 threshold 설정 가능
            - 재발 방지 구조 확보

---

1. 배포 구조
    1. 데이터 수집 → 전처리 → 모델 추론 → anomaly score 계산 → 자동 판정 → 로그 저장
    2. 모델: ONNX 변환 후 Edge 배포
    3. 추론 결과 + 센서 로그 → 서버 저장
2. 모니터링
    - 실시간 anomaly score 분포 모니터링
    - 오탐/미탐 로그 자동 수집
    - 데이터 drift 탐지 (score mean/variance)
        - Score 기반 drift 모니터링은 모델을 평가하는 방법이 아니라,운영 안정성을 보장하는 안전장치입니다. “운영 환경에서는 정답 라벨이 없기 때문에, anomaly score의 mean과 variance를 기반으로 데이터 drift를 모니터링했습니다. 이를 통해 모델 성능이 실제로 떨어지기 전에 threshold 조정이나 재학습 트리거를 걸 수 있도록 설계했습니다.”
        - 📊 데이터 Drift 탐지 – Anomaly Score Mean / Variance 기반 모니터링
        - 1️⃣ 데이터 Drift란?
            - 모델 입력 또는 출력 분포가 학습 시점과 달라지는 현상으로, 모델 성능 저하를 사전에 감지하기 위해 반드시 모니터링해야 하는 요소입니다.
        - Global CS 관점에서는 👉 **성능 저하가 발생한 뒤가 아니라, 발생 “이전”에 감지**하는 게 핵심.
        - 2️⃣ 왜 Score 기반 Drift를 보는가?
            - 제조 현장에서는: GT(정답 라벨) 지연 또는 부재 / 실시간 성능 측정 불가
            - 👉 그래서 **모델 출력 자체(anomaly score)**를 관측하는 방식이 현실적
        - 3️⃣ Mean / Variance 기반 Drift 탐지 개념
            - ✔ 기본 아이디어
            - 학습/검증 시점의 anomaly score 분포: 평균(μ₀) 분산(σ₀²)
            - 운영 중 수집되는 score 분포: 이동 평균(μₜ) 이동 분산(σₜ²)
            - 이 둘의 **차이를 지속적으로 비교**
        - 4️⃣ 실제 모니터링 방식 (운영 관점)
            1. Sliding Window 설정
                - 최근 N개 샘플 (예: 최근 1시간 / 1일)
                - 장비별, 공정별 분리 가능
            2. 통계량 계산
                - Score Mean Drift
                    
                    $|μₜ - μ₀| > α \cdot σ₀$
                    
                - Score Variance Drift
                    
                    $\frac{σₜ²}{σ₀²} > β$
                    
                - 📌 α, β는 운영 안정성 기준으로 설정
            3. Drift 유형 해석
                
                
                | 현상 | 해석 |
                | --- | --- |
                | Mean ↑ | 전반적으로 이상 점수 상승 (환경 악화, 조명 변화 등) |
                | Variance ↑ | 데이터 다양성 증가 (신규 패턴 유입) |
                | Mean ↓ | 입력 분포 변화 또는 모델 과적합 의심 |
        - 5️⃣ Drift 발생 시 대응 전략 (이게 핵심)
            - 🔁 단계적 대응
                1. **Soft Alert**
                    - 운영자 알림
                    - 공정/장비 상태 점검
                2. **Threshold 재조정**
                    - 임시 threshold 상향/하향
                3. **데이터 수집 플래그**
                    - Drift 구간 데이터 자동 저장
                4. **Retraining Trigger**
                    - Drift 지속 시 재학습 파이프라인 실행
        - 6️⃣ 장점 (왜 이 방식을 썼는가)
            - ✔ 라벨 없이 가능
                - 제조 현장에 매우 적합
            - ✔ 실시간 감지 가능
                - 성능 저하 전에 조치 가능
            - ✔ 구현 비용 낮음
                - 통계량 계산만으로 운영 가능
        - 7️⃣ 한계 및 보완 (이 말 하면 신뢰도 ↑)
            - 한계 : Mean/Variance는 복잡한 분포 변화 감지 한계
            - 보완 : KS test, PSI, Wasserstein distance 병행 / 장비별 baseline 분리
                - Mean·Variance는 가볍고 빠르지만, **분포 형태 변화**까지는 한계가 있음 → 그래서 **통계적 분포 비교 지표를 병행**
                1.  KS Test (Kolmogorov–Smirnov Test)
                    1. “운영 중 anomaly score 분포와 학습 시 분포 간 KS Test를 주기적으로 수행하여,
                        
                        p-value가 임계값 이하로 떨어질 경우 drift로 판단했습니다.”
                        
                    
                    > 두 분포의 누적분포함수(CDF) 차이의 최대값을 측정
                    > 
                    > 
                    > $D = \sup_x |F_{train}(x) - F_{prod}(x)|$
                    > 
                    - 언제 쓰는가
                        - anomaly score 분포 / 센서 로그 단일 채널 / 연속형 변수
                    - 장점
                        - 비모수적 (분포 가정 없음) / 구현 간단 / 임계값 설정 명확 (p-value)
                    - 한계
                        - 샘플 수에 민감 / 다변량에는 직접 적용 어려움
                2. PSI (Population Stability Index)
                - “PSI를 활용해 장비별, 공정별 데이터 안정성을 수치화하여 관리했습니다.”
                    
                    > 학습 분포 대비 운영 분포의 변화량을 구간 단위로 측정
                    $PSI = \sum_i (p_i - q_i) \ln \frac{p_i}{q_i}$
                    > 
                    - pᵢ: 학습 데이터 비율
                    - qᵢ: 운영 데이터 비율
                - 언제 쓰는가
                    - 제조 지표 / 모델 score 모니터링 / 장비별 비교
                - 기준값 (실무에서 중요)
                    
                    
                    | PSI | 해석 |
                    | --- | --- |
                    | < 0.1 | 안정 |
                    | 0.1 ~ 0.25 | 경미한 drift |
                    | > 0.25 | 심각한 drift |
                - 장점
                    - 해석 직관적 / 리포트/운영 보고에 매우 적합
                - 한계
                    - binning 방식에 민감 / 미세한 분포 변화에는 둔감
                1. Wasserstein Distance (Earth Mover’s Distance)
                    
                    > 한 분포를 다른 분포로 옮기는데 필요한 최소 비용
                    > 
                - 언제 쓰는가
                    - 평균은 같지만 shape가 다른 경우 / gradual drift 감지 / score 분포 전체 비교
                - 장점
                    - 분포 형태 변화에 매우 민감 / 직관적 해석 가능
                - 단점
                    - 계산 비용 높음 / 실시간 모니터링에는 부담
                - 운영 전략
                    - 실시간: Mean/Variance
                    - 주기 분석: Wasserstein
                - 📌 Drift 지표 병행 전략 (정리)
                    
                    
                    | 목적 | 사용 지표 |
                    | --- | --- |
                    | 실시간 감지 | Mean / Variance |
                    | 통계적 검증 | KS Test |
                    | 안정성 리포트 | PSI |
                    | 분포 형태 변화 | Wasserstein |
        
        ---
        
3. 재학습 전략
    - 조건 기반 retraining
        - 오탐률 상승 / 신규 공정 투입 / 분포 변화 감지 시
    - Human-in-the-loop
        - 현업 검증 데이터 재라벨링
    - 주기적 모델 검증 후 재배포
    - 📌 **장점**

        - 현장 변화에 지속 대응 가능

    
    1. 운영환경:
        - 모델 Serving → Docker/GPU
        - 모니터링: Log 기반 성능/Drift
            
            ✔ QA 자동화: CI/CD + 테스트 모듈
            
    2. 자동화 파이프라인
        - CI 기반 모델 테스트 자동화
        - 성능 저하 시 재학습 트리거
        - 로그 기반 품질 모니터링 대시보드 구성
        
        👉 **검증과 운영까지 포함한 AI Test 플랫폼 관점**

- Data Drift vs Concept Drift 차이
    - “Data Drift는 입력 분포 변화로 통계 지표로 사전 감지가 가능하지만, Concept Drift는 불량 정의나 기준이 바뀌는 경우라 현업 피드백과 성능 저하를 통해 인지하고 모델 자체를 재설계해야 합니다.”
    - 🔹 Data Drift (Input Drift)
        
        > 입력 데이터 분포가 변함, 하지만 **정답 규칙은 동일**
        > 
        - 예시 (제조/비전)
            - 조명 변화
            - 카메라 위치 변경
            - 센서 노이즈 증가
            - 신규 장비 투입
        - 영향 : 모델 입력이 낯설어짐 → score 분포 변화
        - 대응
            - 데이터 보강
            - normalization 재설계
            - threshold 조정
            - 재학습(선택)
    - 🔹 Concept Drift (Label Drift)
        
        > 입력–정답 관계 자체가 변함
        > 
        - 예시
            - 불량 정의 변경
            - 공정 기준 변경
            - 고객 요구 스펙 변경
            - 새로운 결함 유형 등장
        - 영향
            - 기존 모델 논리 자체가 무효화
            - score는 정상인데 실제 불량
        - 대응
            - 라벨 재정의
            - 모델 구조/목적 수정
            - 재학습 필수
    - 🔥 핵심 차이 요약
        
        
        | 구분 | Data Drift | Concept Drift |
        | --- | --- | --- |
        | 변하는 것 | 입력 분포 | 입력-정답 관계 |
        | 라벨 | 유지 | 변경 |
        | 탐지 | 통계적 | 성능/현업 피드백 |
        | 대응 | 보정/재학습 | 재설계 |

### 2.2. VLM 문제 예시

- VLM이 나오는 워크샘플링 문제의 전형적 형태
    
    > 이미지 기반 검사 시스템에서
    > 
    > - 불량 유형이 자주 바뀌고
    > - 명확한 라벨 정의가 어렵고
    > - 현업의 주관적 판단이 개입되는 문제가 있습니다.
    > 
    > 이때 **VLM을 활용한 검사/판정 시스템을 어떻게 설계하겠습니까?**
    > 
    
    또는
    
    > 테스트 결과를 사람이 최종 판정해야 하는 상황에서
    > 
    > 
    > **AI 기반 자동 판정을 고도화하는 방법을 제안하시오.**
    > 
- 문제
    
    > 제조 공정 테스트 과정에서 제품 이미지와 테스트 로그를 기반으로 **Pass / Fail을 자동 판정하는 시스템**을 구축하려고 한다.기존 Vision 기반 Defect Detection 모델은
    
    - 불량 유형이 자주 변경되고
    - 경계가 모호한 불량(경미한 스크래치, 일시적 오동작 등)에 대해 오탐이 증가하는 문제가 있다.
    
    이 문제를 해결하기 위해 **VLM(Vision-Language Model)을 활용한 판정 시스템**을 설계하려 한다.
    아래 항목에 대해 설명하시오.
    
    1. 기존 방식의 한계와 VLM을 사용하는 이유
    2. 데이터 구성 방식 (입력 형태)
    3. VLM 기반 판정 모델 구조
    4. 학습 또는 활용 전략
    5. 성능 평가 지표 및 실험 설계
    6. 운영 환경 배포 및 모니터링 방안
    > 
- 풀이

VLM 문제의 핵심 평가 포인트 (면접관 시선)

❌ “CLIP / LLaVA / GPT-4V 써서 멀티모달로 처리했습니다”

⭕ **“기존 rule/vision-only로 안 되던 판정 문제를 어떻게 구조적으로 해결했는가”**

> VLM = 새로운 분류기 ❌ → **VLM = 판단 로직을 유연하게 만드는 ‘판정 레이어’** ⭕
> 

---

- 3️⃣ 답변 전체 구조 (이대로 쓰면 됨)
    - ✅ VLM 문제 답변의 정석 구조
        
        ```
        1. 기존 방식의 한계
        2. VLM을 쓰는 이유 (왜 필요한가)
        3. 데이터 구성 방식
        4. VLM 기반 판정 구조
        5. 학습 / 튜닝 전략
        6. 검증 및 지표
        7. 운영 시 장점과 리스크
        ```
        
- 4️⃣ 실제 답변 예시 (실전용)
    1. 기존 방식의 한계
    
    기존 Vision 기반 Defect Detection은
    
    - 불량 유형이 고정되어 있어야 하고
    - 경계가 모호한 경우 오탐이 증가하며
    - “이상해 보인다 / 문제 없다”와 같은 **정성적 판단을 반영하기 어렵다는 한계**가 있습니다.
    
    특히 제품 테스트 판정에서는
    
    - *결과 자체보다 ‘**왜 Fail인지 설명 가능해야 한다’**는 요구가 큽니다.

---

1. VLM을 사용하는 이유
    
    VLM은 이미지 특징과 언어 의미를 함께 표현할 수 있어
    
    - 불량을 **고정 클래스가 아닌 ‘상태 설명’으로 판단**할 수 있고
    - 규칙 기반으로 표현하기 어려운 조건을 자연어 기준으로 확장할 수 있습니다.
    
    따라서 VLM을 **최종 판정 레이어**로 활용하는 것이 적합하다고 판단했습니다.
    
2. 데이터 구성 방식
    - 입력 데이터
        - 제품 이미지 (또는 영상 프레임)
        - 센서 로그 요약 정보 (통계량 or 이벤트 플래그)
        - 테스트 시나리오 설명 (TC 문장)
    - 텍스트 프롬프트 예
        
        > “이 이미지는 정상 동작 상태인가, 아니면 기능 오동작이 의심되는가?”
        “표면 스크래치가 기능적 불량으로 판단될 수준인가?”
        > 
        
        👉 **VLM이 ‘판정 기준’을 이해하도록 데이터 구성**
        
3. VLM 기반 판정 구조
    
    ```
    [Vision Encoder] → 이미지 특징
    [Log Encoder] → 센서 요약 정보
    [Text Prompt] → 테스트 기준
            ↓
    [VLM]
            ↓
    - Pass / Fail 확률
    - Fail 사유 요약
    ```
    
    - Vision 모델은 **Defect 후보 영역 추출**
    - VLM은 해당 결과를 **자연어 기준으로 최종 판정**
    
    👉 Vision = 탐지 👉 VLM = 판단
    
4. 학습 / 튜닝 전략
    - 대규모 파인튜닝보다는
        - 제로, Few-shot prompt tuning
        - 테스트 기준 문장 정제
    - 오탐이 중요한 경우
        - “Fail로 판단해야 하는 조건” 중심 프롬프트 설계
    - 🔑 VLM을 파인튜닝할지, 어떻게 사용할지에 대한 답변 구조
        
        **“언제 파인튜닝하고, 언제 안 하는지”를 명확한 기준으로 말할 수 있어야 해.**
        
        - 한 줄 요약 (먼저 말하면 매우 좋음)
            
            > “초기에는 파인튜닝 없이 프롬프트 기반으로 사용하고,판정 기준이 고정되고 데이터가 누적되면 제한적으로 파인튜닝을 고려하겠습니다.”
            > 
        
        ---
        
        - 1️⃣ 기본 전략: 파인튜닝은 ‘최후의 수단’
            
            Global CS센터 업무 특성상:
            
            - 불량 기준이 자주 바뀌고 / 제품/모델/라인이 다양하며 / 운영 안정성이 중요
            
            👉 **대규모 파인튜닝은 초기 전략으로 부적합**
            
            그래서 기본 전략은:
            
        - ✅ 프롬프트 기반 활용 (Zero/Few-shot)
            - 불량 판정 기준을 **자연어로 외부에서 제어**
            - 코드 수정 없이 기준 변경 가능
            - 신규 제품/라인 대응이 빠름
            
            예:
            
            > “기능에 영향을 주는 결함만 Fail로 판단하라”
            “외관 결함은 사용성에 영향을 줄 경우만 Fail로 판단하라”
            > 
        
        ---
        
        - 2️⃣ 그럼 VLM은 어떻게 ‘사용’하나요?
            - ✔ 역할 분리 전략 (매우 중요)
            
            | 컴포넌트 | 역할 |
            | --- | --- |
            | Vision 모델 | 결함 후보 탐지, Feature 추출 |
            | 센서 기반 모델 | 이상 신호 정량화 |
            | **VLM** | 최종 Pass / Fail 판정 + 이유 설명👉 VLM은 **“의사결정 레이어”**로 사용 |
            
            👉 탐지 자체는 기존 Vision/ML이 담당
            
            이렇게 하면:
            
            - 성능 불안정 리스크 감소
            - 설명 가능성 증가
            - 운영 제어 용이
        
        ---
        
        - 3️⃣ 그럼 언제 파인튜닝을 고려하나요?
            - 🎯 파인튜닝을 고려하는 조건 (명확히 말해야 함)
            1. 판정 기준이 안정화되었을 때
                - 특정 공정/제품에서 Fail 기준이 고정됨
                - 현업 합의가 끝난 상태
            2. 실패 케이스가 반복적으로 발생할 때
                - 특정 유형에서 지속적인 오판
                - 프롬프트만으로 한계가 명확할 때
            3. 충분한 로그/판정 데이터가 누적되었을 때
                - 이미지 + 판정 결과 + 이유 텍스트
                - 운영 환경 데이터 확보
        
        ---
        
        - 4️⃣ 파인튜닝을 한다면 “어디까지” 할 것인가?
            
            ❌ 전체 모델 파인튜닝 → 리스크 큼

                - **LoRA / Adapter 기반 미세조정**

                - 특정 제품군 또는 공정 단위로 분리 적용
                
                👉 공정 A용, 공정 B용 모델을 분리 운영 가능
                
        
        ---
        
        - 5️⃣ 파인튜닝의 장단점 인식 (이걸 말하면 진짜 잘함)
            - 👍 장점
                - 특정 도메인 판정 정확도 상승
                - Fail 기준 일관성 강화
            - ⚠️ 단점
                - 기준 변경 시 재학습 필요
                - 운영 복잡도 증가
                - Drift 발생 시 성능 급락 가능
                
                👉 그래서 **프롬프트 → 경량 파인튜닝 → 운영 평가**
                
                👉 단계적 접근이 필수

### 2.3. 🔤 CI / CI 기반 모델 테스트 / CD / QA 자동화 정리

---

- 1️⃣ CI (Continuous Integration)
    - 🔹 약어 의미
        - **CI = Continuous Integration (지속적 통합)**
    - 🔹 기본 뜻
        - 코드나 모델 변경 사항을 **자주(지속적으로) 합쳐서** 문제가 없는지 **자동으로 검증**하는 과정
    - 🔹 AI/모델 관점 해석
        
        > “모델이나 추론 코드가 바뀔 때마다자동으로 테스트를 돌려보는 단계”
        > 
    - 🔹 예시
        - 모델 가중치 변경
        - 전처리 코드 수정
        - threshold 값 변경
        - → **자동 테스트 실행**

---

- 2️⃣ CI 기반 모델 테스트 (Model Testing in CI)
    - 🔹 약어 의미
        - **CI 기반 모델 테스트는 ‘CI 단계에서 수행하는 모델 검증’**
    - 🔹 기본 뜻
        - 모델 변경 시 사람이 직접 테스트하지 않고 **자동으로 성능/안정성을 검증**
    - 🔹 AI/검증 관점 핵심
        
        > “모델 성능이 예전보다 나빠지지 않았는지 자동 확인”
        > 
    - 🔹 대표 테스트 항목
        - Precision / Recall / F1 변화
        - Pass/Fail 비율 변화
        - score 분산 증가 여부
        - latency 증가 여부
        
        👉 **성능 회귀(Regression) 방지 목적**
        

---

- 3️⃣ CD (Continuous Deployment / Continuous Delivery)
    - 🔹 약어 의미
        - **CD = Continuous Deployment 또는 Continuous Delivery**
        
        > 회사마다 Deployment / Delivery 용어 혼용함
        > 
    - 🔹 기본 뜻
        - CI로 검증이 끝난 결과를 **운영 환경에 반영하는 단계**
    - 🔹 AI/모델 관점 해석
        
        > “테스트를 통과한 모델을운영 시스템에 적용하는 단계”
        > 

---

- 4️⃣ QA 자동화 (Quality Assurance Automation)
    - 🔹 약어 의미
        - **QA = Quality Assurance (품질 보증)**
    - 🔹 기본 뜻
        - 사람이 수동으로 하던 검증 작업을 **자동화된 테스트로 수행**
    - 🔹 AI/검증 관점 핵심
        
        > “모델이 항상 같은 기준으로, 안정적으로 동작하는지 자동으로 확인”
        > 
    - 🔹 QA 자동화 예
        - 모델 성능 체크 자동화
        - 판정 기준 검증
        - 실패 케이스 재현 테스트
        - 테스트 결과 리포트 자동 생성
    
    ---
    
    - 🔁 전체 흐름 한 번에 이해하기
        
        ```
        [모델/코드 변경]
                ↓
        CI → “변경될 때마다 자동으로 검증하는 과정”
        (자동 테스트 실행)
                ↓
        CI 기반 모델 테스트 → “모델 성능이 떨어지지 않았는지 자동 확인”
        (성능/안정성 검증)
                ↓
        CD → “검증된 모델을 운영에 반영하는 단계”
        (운영 환경 반영)
                ↓
        QA 자동화→ “사람 대신 시스템이 품질을 지속적으로 확인”
        (지속적인 품질 감시)
        ```

안녕하세요, 저는 **AI 모델을 연구로 끝내지 않고, Vision 및 Multimodal 기반으로 실제 제품과 운영 환경에 적용되는 시스템을 만든 경험**이 있습니다.

저는 모델 성능 자체보다 ‘현장에서 어떻게 동작하고 검증되는가’에 집중해왔습니다. 웹 개발자로 커리어를 시작하며 사용자 관점에서 서비스가 어떻게 사용되는지를 고민했고, 이후 AI 분야로 전향하면서도 이 관점을 그대로 가져와 연구와 실무를 이어왔습니다.

LG전자에서 Vision 기반 행동 인식 모델을 Edge 환경에 최적화 학습했고, 가전 로그·영상·오디오 데이터를 결합한 Multimodal 상황 인식 PoC를 수행하며 데이터 수집–전처리–모델링–검증–배포까지 파이프라인을 연결했습니다. 특히 모델 오판정과 실패 케이스를 줄이기 위한 데이터 정의와 실시간 판정 구조를 설계한 경험은 Global CS센터의 업무와 직접적으로 맞닿아 있다고 생각합니다.

AI를 ‘잘 만드는 것’에서 그치지 않고, 검증 가능하고 운영 가능한 형태로 만드는 역할을 수행해 왔고, 이 경험을 바탕으로 Global CS센터의 AI Vision 및 데이터 최적화 업무에 바로 기여할 수 있다고 생각해 지원하게 되었습니다.

### **Q9. 내 강점이 G-CS 직무에 어떻게 맞는지**

- **강점**

제 강점은 문제를 구조화하고 AI 모델을 **‘검증 가능한 시스템’**으로 만드는 **경험**과 **끝까지 완결시키는 실행력**입니다.

연구 단계에서 성능이 잘 나와도, 실제 환경에서는 데이터 노이즈나 예외 상황으로 문제가 발생하는 경우가 많았고, 이를 해결하기 위해 **데이터 품질 정의, 로그 분석, 운영 기준 수립까지 함께 고민**해왔습니다.

Vision 모델 개발뿐 아니라, 대규모 데이터 전처리와 라벨링 전략 설계, 실패 case 분석을 통한 데이터 정제 및 성능 개선을 반복해왔고, Multimodal 프로젝트에서는 영상·로그·음성 데이터를 의미 단위로 정제하고, 특히 edge 환경에서의 제약, 오탐지 및 미탐지 관리, 운영 중 데이터 품질 문제를 직접 겪어본 경험은 Global-CS에서 요구하는 모델 테스트 및 검증, 자동 판정, 운영 환경 배포와 잘 맞는다고 생각합니다.

- **약점** 본인의 단점 한 가지를 말해본다면? **(보완 방식 포함)**

“초기에는 문제를 너무 깊이 파고들다 일정 관리가 어려웠던 경험이 있습니다.

이후에는 초반에 ‘최소 검증 목표(MVP 기준)’를 명확히 정하고, 단계별 목표를 나누어 관리하는 방식으로 개선했습니다.

현재는 깊이와 속도의 균형을 맞추는 데 훨씬 익숙해졌습니다.”

- **비슷한 질문 - 내가 지원 직무에 적합한 이유 (약 1분)**

제가 잘할 수 있는 비전 분야이기도 했구요, Global CS센터 **직무**는 AI 모델을 “잘 만드는 것”뿐만 아니라 **제품 품질을 안정적으로 검증하고, 문제를 조기에 감지하며, 자동화 구조로 확산시키는 역할이 핵심**이라고 이해하고 있습니다.

저는 Vision 기반 인식 모델을 실제 제품에 적용하며, 데이터 정의 → 모델 학습 → 성능 분석 → 운영 환경 배포 → 문제 재현 및 개선까지 전 과정을 경험했습니다. 특히 가전 로그·영상 기반 상황 탐지 PoC에서는 **모델 성능보다 판정 기준, 임계값 설정, 실패 케이스 분석이 훨씬 중요**하다는 점을 체감했습니다.

이러한 경험은 제조·제품 품질 데이터 기반 불량 예측, Vision 검사, Test 자동화 플랫폼 구축이라는 본 직무와 직접적으로 맞닿아 있다고 생각해서 지원하게 되었습니다.

### Q10. 성공 경험 & 실패 케이스

실패했던 케이스를 바탕으로 성공 경험으로 개선했던 케이스를 설명드리겠습니다.

멀티모달 이벤트 감지 PoC를 진행할 때 초기 학습 데이터에서는 성능이 우수했지만 실제 제품 환경에서는 빠른 움직임과 조명 변화로 오인식이 발생했고, 사용자의 행동을 미리 인식해 가전을 제어해야 하는 시점에서 인식 지연이 발생한 경험이 있습니다. 처음에는 모델 구조나 학습 방식의 문제라고 판단했지만, 분석 결과 **데이터 분포 차이와 실패 케이스 정의가 불명확했던 점**이 핵심 원인이었습니다.

이를 해결하기 위해 먼저 **failure case를 명확히 정의해서** 단순 정확도가 아닌, 실제 서비스 관점에서 중요한 시나리오를 기준으로 **오탐·미탐 유형을 세분화**했습니다. (예를 들면 귀가/외출, 조명이 변화하여 이벤트를 감지하지 못하는 구간, 빠른 방향 전환 등 모델이 헷갈려하는 유형이 있었습니다.) 이후 도메인 특화 데이터 보강을 위해 귀가 또는 외출 상황에서 더 긴 temporal context를 반영하도록 시계열 데이터를 재구성하고, 조명 변화·가림 현상이 빈번한 구간을 중심으로 데이터를 증강했습니다.

또한 **가전 제어 관점에서의 인식 타이밍 문제**를 해결하기 위해, 행동을 완전히 확정한 이후 제어하는 방식이 아니라 **행동 가능성이 일정 threshold 이상일 때 가전을 선제적으로 동작시키는 구조**로 정책을 수정했습니다. 예를 들어, 사용자가 귀가 동작의 중간 단계에 진입했을 때 **temporal window를 재설계하여** Vision 기반 행동 인식 결과를 **시간 단위로 누적 판단하는 로직**으로 변경하고, 단일 프레임 오인식이 바로 제어로 이어지지 않도록 **confidence 기반 판정 구조**를 설계했습니다.

이 경험을 통해 저는 실패를 단순한 모델 성능 문제로 보지 않고, **데이터 정의·검증 기준·운영 정책을 함께 설계해야 AI가 실제 제품에서 신뢰받을 수 있다**는 관점을 갖게 되었습니다.

- **자동 평가 방식 물어보면**

평가 방식에서도 기존 단일 정확도 지표 대신, **행동 인식 정확도, 인식 시점 지연(latency), 오탐/미탐률을 분리한 자동 평가 지표 체계**를 설계했습니다. 이를 통해 모델 성능과 사용자 체감 품질을 동시에 관리할 수 있었고, 결과적으로 안정성과 실사용 만족도를 함께 개선할 수 있었습니다. ### **Q11. LG전자 → Global CS센터로 이직하려는 이유 (가장 중요 / 약 1~1.5분)**

- **왜 삼성전자로 오려고 하는가?**

LG전자에서 Vision·Multimodal AI 모델을 연구하고 제품에 적용하며 많은 성장을 할 수 있었습니다. 하지만 경험을 쌓을수록 **AI 모델 자체보다, 검증·운영·품질 판단 구조의 중요성을 더 크게 느끼게 되었습니다**.

Global CS센터는 AI를 활용해 각 사업부의 다양한 제품 기능을 검증하고, 품질 문제를 데이터 기반으로 정의·자동화하는 조직으로 알고 있습니다. 이는 제가 연구와 실무를 통해 가장 잘할 수 있고, 더 깊게 성장하고 싶은 방향이라고 생각이 되었습니다.

특히 Test 자동화, AI 판정, 운영 환경 배포까지 책임지는 구조 속에서 제가 쌓아온 Vision·Multimodal·Edge 경험을 제품 품질과 고객 신뢰를 직접 높이는 결과로 연결하고 싶어 지원하게 되었습니다.

- **추가질문 + “모델보다 검증·운영·품질 판단 구조의 중요성을 느낀 계기”**

계기는 Vision 및 Multimodal AI 모델을 개발하며, 경험이 쌓일수록 모델 자체보다 검증·운영·품질 판단 구조의 중요성을 더 크게 느끼게 되었습니다.

예를 들어, 가전 로그·영상 기반 상황 인지 모델을 PoC로 구현했을 때, 모델 정확도는 기준을 만족했지만 실제 사용 환경에서는 미검출·오탐이 사용자 경험과 CS 이슈로 직결될 수 있음을 확인했습니다. 이 과정에서 단순 성능 개선보다,

“어떤 조건에서 문제로 판단할 것인지”,

“어떤 경우를 정상/비정상으로 볼 것인지”,

“이를 자동으로 검증하고 재현할 수 있는 구조가 있는지”가 훨씬 중요하다는 것을 체감했습니다.

Global CS센터는 이러한 AI 기반 검증·판정·자동화 구조를 전사적으로 설계하고 확산하는 조직이라고 이해하고 있으며, 제가 쌓아온 경험을 가장 잘 활용할 수 있는 곳이라고 판단해 지원하게 되었습니다.

### **Q12. 회사생활**

✅ 1️⃣ 기본형 (가장 안전하고 정석적인 답변)

> 저는 회사생활을 개인의 성과보다 팀과 조직의 목표를 달성하는 과정이라고 생각합니다.
> 
> 
> 지금까지 연구실, 산학 과제, 회사에서 일하며 혼자 잘하는 것보다 **공통 목표를 명확히 하고 역할을 나누는 것**이 훨씬 중요하다는 것을 배웠습니다.
> 
> 특히 문제 상황이 생겼을 때 혼자 해결하려 하기보다는, 상황을 정리해 공유하고 함께 해결 방향을 찾는 방식이 결과적으로 더 빠르고 안정적인 성과로 이어진다고 생각합니다.
> 
> 맡은 역할에 대해서는 끝까지 책임지고, 필요한 경우에는 제 역할 범위를 넘어가더라도 팀에 도움이 되는 방향으로 움직이려 노력해왔습니다.
> 

👉 **“조직 적응력 + 책임감”**을 동시에 보여주는 답변

---

✅ 2️⃣ Global CS센터에 딱 맞춘 답변 (운영·검증 조직 톤)

> 회사생활은 단순히 기술을 개발하는 것이 아니라, 현장에서 실제로 쓰일 수 있도록 안정성과 신뢰성을 함께 만들어가는 과정이라고 생각합니다.
> 
> 
> 회사에서도 모델 성능 자체보다 운영 환경, 실패 케이스, 검증 기준을 함께 고려하지 않으면 실제 서비스에서 문제가 발생한다는 경험을 했습니다.
> 
> 그래서 저는 회사생활에서 중요한 것은 기술적인 주장보다 **왜 문제가 발생했는지 설명하고, 운영 관점에서 공감 가능한 해결안을 제시하는 태도**라고 생각합니다.
> 
> 이러한 관점으로 개발, 검증, 운영 조직과 소통하며 일해왔고, Global CS센터에서도 같은 자세로 팀에 기여하고 싶습니다.
> 

👉 **“이 사람은 연구형이 아니라 CS 조직에 잘 맞는다”**는 인상을 줌

> 
> 
> 연구실과 회사에서 비교적 막내 역할을 오래 하며, 먼저 배우고 질문하고 피드백을 받아들이는 태도를 자연스럽게 익혔습니다.
>
