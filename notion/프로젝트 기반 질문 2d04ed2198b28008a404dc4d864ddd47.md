### **✅ 프로젝트 기반 기술 질문: Examodal**

- **Q1. Examodal 프로젝트에서 가장 기술적으로 어려웠던 부분은 무엇이었고, 어떻게 해결하셨나요?**
    
    **[상황]** Examodal 프로젝트에서는 이미지와 오디오 모달리티를 기존 LLM에 통합하는 과정에서 modality 간 정합성과 학습 안정성 확보가 가장 큰 과제였습니다.
    
    **[문제]** 특히 오디오 입력을 기존 텍스트 중심 LLM에 자연스럽게 통합하려 할 때, catastrophic forgetting 문제가 발생했고, modality fusion의 순서와 방법에 따라 성능이 크게 달라졌습니다.
    
    **[해결]** 이 문제를 해결하기 위해 **cross-modal continual learning 전략**을 도입했습니다. Merge then realign 이란 논문이나 `1` 최근 논문들에서 reverse order 보다 sequential order가 학습이 잘된다는 점을 참고하여 학습 순서를 modality-wise로 image, audio, video 로 구성하고 `2` 두 단계를 반복으로 학습했습니다. 먼저 새 모달리티를 학습한 모델을 기존 모델과 cumulative moving average merge 방식을 활용해 합치고 replay buffer를 활용한 rehearsal 방식을 적용하는 두번째 단계를 통과시켰습니다. 이 과정을 4번정도 반복하며 forgetting 방지 96% 정도 기존 모달리티 성능 유지하며 오디오 성능 굿
    
    <aside>
    🔍
    
    - **문제: Catastrophic Forgetting**
        
        멀티모달 LLM에서 새로운 모달리티(예: 오디오)를 학습시킬 때, 기존에 잘 학습된 텍스트/이미지 성능이 급격히 저하되는 현상을 catastrophic forgetting이라고 합니다. 새로운 데이터 분포에 모델이 과적합되면서, 이전 지식을 “잊는” 현상입니다.
        
    - **해결 전략: Rehearsal (복습 학습)**
        
        이를 방지하기 위해 rehearsal strategy를 씁니다. 즉, 새로운 데이터를 학습할 때 이전에 학습한 데이터 일부도 같이 보여주면서 학습시키는 방식이죠.
        
    - **Replay Buffer란?**
        
        Replay Buffer는 과거 학습 데이터의 일부를 저장해두었다가, 나중에 새로운 학습 중간중간에 같이 꺼내서 다시 학습에 사용하는 메커니즘입니다.
        
        원래는 강화학습에서 많이 쓰였지만, continual learning에서도 대표적인 anti-forgetting 기법이에요.
        
    
    **Rehearsal란?**
    
    “아, rehearsal은 continual learning에서 자주 쓰이는 전략인데요.
    
    새로운 모달리티나 task를 학습할 때, 기존에 배운 내용을 잊어버리는 문제를 막기 위해 예전 데이터를 일부 같이 학습에 넣는 방식이에요.
    
    저는 Examodal에서 오디오 모달리티를 확장할 때, 기존 텍스트/이미지 성능 저하를 막기 위해 replay buffer 기반의 Exact Rehearsal 방식을 실험했어요.
    
    실제로 buffer에 일부 대표 샘플을 저장해두고, 오디오 학습 도중 섞어서 같이 학습하니까 forgetting 현상이 확실히 줄어드는 걸 확인할 수 있었습니다.”
    
    - **종류**
        1. **Exact Rehearsal (정확 리허설)**
        - 이전 task의 데이터를 그대로 저장하고, 나중에 새 task 학습할 때 일부 같이 학습.
        - Replay Buffer 방식이 여기에 해당합니다.
        - 가장 효과는 좋지만, 메모리 사용량이 커질 수 있어요.
        1. **Pseudo Rehearsal (가짜 리허설)**
        - 예전 데이터를 저장하는 대신, 이전 모델이 만든 예측값을 가짜 데이터로 사용.
        - 예: 이미지 분류 모델이 예전에 “고양이”라고 분류했던 이미지를 기억하고, 그 label만 유지하며 다시 학습.
        - 메모리는 적게 들지만 성능은 상대적으로 떨어질 수 있습니다.
        1. **Generative Rehearsal**
        - 과거 데이터를 저장하지 않고, *생성 모델(GAN, VAE 등)*을 이용해 이전 task와 유사한 synthetic data를 만들어 replay함.
        - ex) text generation model을 사용해 과거 task의 텍스트를 다시 생성
    
    **Knowledge distillation 질문 나오면?**
    
    `1` 대학원 시절 teacher student 기반 네트워크 학습을 해보았고,`2` 저희 팀에서 최근 siglip 기반으로 MLM을 학습한 뒤에 더 작은 비전 인코더인 siglip2를 distillation 하면서 학습했습니다.  ~~→ 이 말 할라면 siglip2 동민님 자료 찾아보기 →~~ siglip 1, 2 의 리샘플러 아웃풋 임베딩을 서로 KL loss 사용해서 align 하지만 이것만으로는 siglip 1 이상의 성능을 내기 어려워 LM loss 도 같이 사용해 성능을 높인 적이 있었다.

- **Q1.4. continual learning 전략에서 사용한 loss 구조나 적용 순서는 어떻게 설정했나요?**
    
    **[목표]** 네, 저는 기존 텍스트/이미지 기반 LLM에 오디오 모달리티를 확장하면서 **기존 성능 저하를 막는 게 핵심 과제**라고 봤습니다. 그래서 기존 이미지/텍스트 기반 LLM에 오디오 모달리티를 확장하는 과정에서
    
    **[학습 순서]** modality 전이를 최소화하기 위해 **텍스트/이미지 기반 모델을 SFT로 먼저 고정**한 후, 오디오 입력을 포함하는 추가 task들을 continual learning 방식으로 순차 학습했는데 이때 이미지/텍스트 관련 샘플 일부를 **replay buffer**에서 계속 섞어줬고, 학습 순서는 `text → image → audio` 순으로 설계했습니다.
    
    **[Loss 구성] `1`** Loss는 동일하게 pretrain loss로는 next token prediction, sft에서는 supervised finetune loss를 사용했지만, continual setting에서는 몇 가지 안정화 요소를 넣었어요. 예를 들면, `2` **현재 모달리티엔 높은 loss weight**, 기존 모달리티엔 낮은 가중치를 줬고, 기존 모델과의 `3` **출력 분포 차이를 줄이기 위해 KL-divergence loss** 도 일부 도입했습니다. 그리고 `4` replay 샘플에는 learning rate를 낮추고 step도 줄여서 **기존 representation을 손상시키지 않도록** 조절했어요.
    
    **[결과]** 그 결과 기존 이미지/텍스트 성능은 유지하면서 오디오 쪽 성능도 안정적으로 올라갔고, pretrain을 다시 하지 않아도 되는 구조가 가능했습니다
    
    - **KL-Loss: 기존 LLM과 출력 분포 차이 줄이기**
        
        🔣 수식 관점에서 설명
        
        - 기존 모델을 **teacher** (혹은 old model), 현재 학습 중인 모델을 **student** (혹은 new model)이라고 할 때,
            
            $$
            L_{KL}=KL(P_{old}(y∣x)∣∣P_{new}(y∣x))
            $$
            
        - `P_old(y|x)`: 기존 LLM이 입력 `x`에 대해 내던 softmax 출력 (예: logit or soft label)
        - `P_new(y|x)`: 새롭게 학습 중인 모델의 현재 출력
        - 이 divergence가 작아질수록, new model은 old model과 비슷한 출력을 내게 됩니다.
        
        > 학습 Loss = Original CE loss + λ * KL divergence loss
        > 
        > 
        > 여기서 `λ`는 regularization 강도를 조절하는 하이퍼 파라미터
        > 
        
        🧪 구현 예시 (PyTorch 기준)
        
        ```python
        import torch.nn.functional as F
        
        # 기존 모델과 새 모델의 logit 출력
        old_logits = old_model(x).detach()  # detach to avoid gradient
        new_logits = new_model(x)
        
        # Softmax 온도 조절 (선택사항)
        T = 1.0
        old_probs = F.softmax(old_logits / T, dim=-1)
        new_log_probs = F.log_softmax(new_logits / T, dim=-1)
        
        # KL divergence loss
        kl_loss = F.kl_div(new_log_probs, old_probs, reduction="batchmean") * (T ** 2)
        ```
        
        이걸 전체 loss에 아래처럼 더합니다:
        
        ```python
        total_loss = ce_loss + lambda_kl * kl_loss
        ```

- **Q2.1. Audio에서 퍼시버 리샘플러를 사용한 이유는 뭔가요?**
    
    “Q-former는 고정된 query 수에 의존하고, linear layer는 피처 평균처럼 정보 손실이 클 수 있습니다. 오디오 입력을 단순하게 linear projection하거나 fixed query(Q-former)로 변환하는 대신, Perceiver Resampler를 통해 오디오의 temporal 정보와 의미 구조를 다층 attention으로 요약하여 context-length의 제약없이 다양한 표현을 활용할 수 있었고, LLM이 더 효과적으로 받아들일 수 있도록 표현력을 강화했다”
    
    $S \geq I(z_{img};z_{txt})$ 
    
    $U_{img} \geq I_{NCE}(z^\prime_{img};u_{img}) + I_{NCE}(z^\prime_{img}z_{txt};u_{img})$
    
    $U_{txt} \geq I_{NCE}(z^\prime_{txt};u_{txt}) + I_{NCE}(z^\prime_{txt}z_{img};u_{txt})$
    
    **🧠 개념 먼저: Perceiver Resampler란?**
    
    - **Perceiver Resampler**는 이미지에서 추출된 **patch-level embeddings (예: 196개)**을
        
        → 소수의 **fixed learnable token**으로 요약(resample)하는 구조입니다.
        
    - 이는 Google의 **Perceiver IO** 구조에서 영감을 받은 것으로,
        
        다양한 길이의 입력을 **정해진 길이의 representation**으로 요약하는 용도로 사용됩니다.
        
    
    **🧱 구성 요소 요약**
    
    | 구성 요소 | 설명 |
    | --- | --- |
    | 📸 Image Patch Features | ViT나 CNN 등을 통해 추출된 이미지 피처 (보통 196개, 768차원 등) |
    | 🧩 Learnable Latents | (예: 32개) 고정된 수의 학습 가능한 벡터 (초기화는 random) |
    | 🔁 Cross-Attention | 이미지 피처를 대상으로 latents가 attention을 통해 정보를 "읽어오는" 방식 |
    | 🧪 Output | 압축된 latent embeddings → LLM의 입력으로 사용 가능 |
    
    **🔄 동작 흐름**
    
    1. **이미지 인코딩**
        - 예: ViT로부터 14×14 grid → 196개의 patch embeddings 생성
    2. **Latent Query Tokens 초기화**
        - 예: 32개의 learnable tokens (형태: `[32, D]`)
    3. **Cross-Attention 적용**
        - 각 latent token이 이미지 피처 전체를 attention 함
        - 즉, Latent token이 “이미지 전체를 요약해서 받아오는 역할” 수행
        - 여러 개의 Transformer block을 거치며 정보를 더 정제
    4. **LLM에 전달**
        - 최종 latent tokens을 LLM의 multimodal prefix로 붙이거나, special token으로 넣어줌
        - 결과적으로 이미지 정보를 **길이 제한 없이 요약된 형태로 LLM이 읽을 수 있게 해줌**
    
    **🔬 SigLIP에서의 Perceiver Resampler의 역할**
    
    SigLIP은 원래 **Contrastive Learning + Image-Text Matching**을 위한 구조지만,
    
    Perceiver Resampler는 다음과 같은 이유로 포함됩니다:
    
    - LLM 연동 목적 (예: Flamingo, PaLI, Gemini 등에서)
    - Text encoder가 기대하는 **고정된 sequence length**와 **정보밀도 높은 representation** 필요
    - Efficient한 multimodal attention을 위해 ViT 출력을 요약하는 방식으로 사용
    
    SigLIP 자체에서는 classification/logit을 위한 부분도 있지만,
    
    **multimodal reasoning task**로 확장하는 구조에서는 Perceiver Resampler가 핵심 구성으로 쓰입니다.
    
    **🔧 코드 스타일 예시 (pseudo)**
    
    ```python
    # input: image_features [B, N, D] (N=196, D=768)
    # learnable_latents [L, D] (L=32)
    
    for _ in range(num_blocks):
        # cross-attn: latent → image
        latents = cross_attention(queries=latents, keys=image_features, values=image_features)
        latents = feedforward(latents)
    ```
    
    **✅ 장점 요약**
    
    | 장점 | 설명 |
    | --- | --- |
    | 길이 압축 | 긴 image patch sequence → 짧은 latent로 요약 |
    | 메모리 효율 | attention 계산량 감소 (O(L×N) vs. O(N²)) |
    | LLM 연동 용이 | LLM의 context length 제한 내에서 처리 가능 |
    | 정보 유지 | Cross-attn 방식으로 이미지의 global context 유지 |
    
    **🤝 관련 구조**
    
    | 모델 | 비슷한 구조 |
    | --- | --- |
    | Flamingo | Perceiver Resampler 사용해 image→text 연결 |
    | PaLI | Vision encoder → MLP projection / attention |
    | BLIP-2 | Q-former로 비슷한 역할 수행 (learnable queries) |
- **Q2.2. 오디오 encoder는 어떤 구조(Whisper base 등)를 사용했나요?**

- **Q2.3. EAT vs BEATs vs Whisper encoder 비교**
    
    오디오 모달리티 인코딩에는 목적에 따라 다양한 인코더를 사용할 수 있는데요, 저희 프로젝트에서는 LLM 통합과 cross-modal reasoning 중심이었기 때문에 **가볍고 fixed-length representation을 생성할 수 있는 EAT 구조**를 사용했습니다. Whisper는 transcription에는 매우 강력하지만, LLM 입력으로 쓰기엔 무겁고 후처리가 필요하고요, BEATs는 시간축 모델링은 좋지만 LLM과 직접 연결하려면 추가 구조가 필요합니다.
    
    반면 EAT는 **CLAP 기반으로 학습된 embedding 구조여서,** **텍스트와 자연스럽게 align되고, LLM에 바로 붙일 수 있다는 점이 장점**이었습니다.
    
- **Q2.4. cross-modal continual learning에서 modality 간 conflict를 어떻게 조절했나요?**
    
    네, cross-modal continual learning 구조에서는 **modality 간 gradient conflict**가 주요 이슈였어요. 저는 오디오 모달리티를 추가하면서 텍스트나 이미지에 대한 성능이 조금씩 떨어지는 걸 확인했고, 이건 shared parameter에 여러 모달리티가 영향을 주면서 발생하는 문제라고 판단했어요.
    
    그래서 우선 학습 순서를 `text → image → audio`로 구성하고, 두단계로 나눠서 학습.
    
    1. 새로운 모달리티에 대한 projector 와 llm 을 학습한 후 기존 모델과 merge하였습니다. 
    2. 이후 이전 모달리티 데이터 일부를 **replay buffer로 유지해서 계속 같이 학습**했어요. 또 loss 계산할 때는 현재 모달리티엔 높은 가중치를, 이전 모달리티엔 낮은 가중치를 줘서 **정보는 유지하되 간섭은 줄이는 구조**를 만들었고 기존 모달리티는 **low LR로 조정**해서, 새로운 모달리티가 기존 표현을 덮어쓰지 않도록 했습니다.
    
    이렇게 하니까 오디오 task는 잘 올라가면서도, 텍스트나 이미지 성능은 거의 떨어지지 않았고, pretrain을 다시 하지 않아도 안정적인 통합이 가능했어요.
    
- **Q2.5. 학습 효율성, 메모리 사용량, latency 측면에서 어떤 성능 개선이 있었나요?**
    
    **[학습 효율성] 이미지 → 오디오 continual learning 개발이 완료된 이후 video 데이터는 full retrain으로 학습이 되었는데,** 이미지, 비디오를 합쳐 1 에폭에 h100 8대 gpu로 120시간 이었는데 retrain 하지 않을 수 있었습니다. SFT 단계에서 오디오 instruction만 별도로 학습해도 이미지 오디오도 96%의 성능을 유지하는 것을 확인했습니다.
    
    **[메모리 사용량]** 1번 단계에서 새로운 모달리티에 대한 projector 와 llm 을 학습한 후 merge하였기 때문에 20-30% 정도 메모리 절약
    
    **[Latency]** inference 시에는 오디오 embedding을 CLIP과 유사한 방식으로 사전 추출하고, 특수 토큰 위치에 삽입하여 처리하므로, 실시간에 가까운 수준(수백 ms 내)에서 이뤄졌습니다.
    
    또한 기존 multimodal fusion보다 attention path가 짧고, token 수를 줄였기 때문에 **latency도 평균 10~15% 감소**했습니다.

- **Q3.1. DPO 적용 시 reward model은 직접 구성했나요, 기존 것을 활용했나요?**
    
    따로 reward model을 사용하진 않고 저렇게 prefer 데이터를 생성해서 사용했습니다.  MMPR 데이터셋 사용해서 dpo 학습 이후 GRPO까지 실험해 본 경험이 있습니다. 카나나-1.5 개발기에서도 이렇게 비슷한 generative reward model을 사용한 거 같은데, 저는 Deepseek R1의 reward 모델을 사용했고 사용한 reward는 accuracy, format, length 리워드가 있습니다. MMPR 데이터셋 260k 기준으로 DPO 대비 6.9% 정도 accuracy 성능 향상이 있는 것을 확인했습니다.
    
    **`실험 1`  acc, format (1:0.5)**
    
    - format reward: Reasoning step을 <think>..</think> 사이에 추가하고 최종 답변을 "Final answer:.." 이후에 하도록 유도
    - Training 중 reward 추세
        - 초반에는 accuracy reward 위주로 학습되고 format reward는 0.0
        - accuracy reward가 거의 1.0에 도달한 후로 format reward 값이 증가
    - 1) <think> tag 없이는 답을 하지 못하고2) completion length가 60대로 줄어듦
    
    **`실험 2`  acc, format, length (1:0.4:0.3)**
    
    - format reward: 최종 답변만 "Final answer:.." 이후에 하도록 유도 (Step 같은 추가 조건도 넣지 않음 → 이 조건에 또 집중할 것으로 예상)
    - Length reward : completion length가 짧아지는 현상을 보완하기 위해 reasoning 과정이 50~200 token 이내가 되도록 제한
        - 50 token 이하 : -1.0
        - 50~200 token : +1.0
        - 200 token 이상 : -0.5
    - Train/Eval Comment
        - Training 중 reward 추세
            - Acc ↓ / Format ↑ 상태로 시작
            - 초반에는 acc 올리면서 format은 떨어지다가 둘 다 올라감
        - Loss 가 크게 줄지 않고, 뒷쪽에 튀는 현상 있음
    - GRPO
        
        **🧠 GRPO란?**
        
        **GRPO (Generative Reward-Preferring Optimization)**는 DeepSeek-VL 팀이 제안한 **generative multimodal reward model을 기반으로 한 preference fine-tuning 전략**입니다.
        
        DPO처럼 “좋은 응답 vs. 나쁜 응답” 쌍을 이용해 학습하지만, DPO와 달리 **explicit한 reward model**을 사용해, 모델이 “왜 이 응답이 더 좋은지”를 generative 방식으로 **학습하고 설명 가능하게** 만드는 게 핵심입니다.
        
        **✅ GRPO 구조 요약**
        
        **1. 📥 Input**
        
        - (Image, Instruction, Chosen Answer, Rejected Answer)
        
        **2. 🤖 Reward Model**
        
        - LLM 구조의 **Generative Reward Model (GRM)** 사용
        - 두 개의 응답(Chosen vs. Rejected)에 대해 **가치 평가 점수(score)**를 생성하거나,
        - **차이의 확률**을 출력하는 구조
        
        **3. 🧪 학습 방식**
        
        - 기존 모델과 reward model 간의 **KL-regularized loss** 기반 학습
        - 목적: "Reward model이 더 높은 점수를 주는 응답을 생성하도록 LLM을 조정"
        
        ## 🔁 DPO vs GRPO 차이
        
        | 항목 | DPO | GRPO |
        | --- | --- | --- |
        | Reward model 사용 | ❌ 없음 (implicit) | ✅ 있음 (explicit) |
        | 목표 | preference 확률 최대화 | reward 점수 기반 개선 |
        | 수식 기반 | preference binary loss (sigmoid) | reward-guided KL loss |
        | 설명력 | 낮음 | 높음 (reward 기준 명확) |
        | 멀티모달 확장성 | 제한적 | 🎯 강력 (image + text 기준 학습 가능) |
        
        **📚 GRPO 수식 구조 (간단 버전)**
        
        GRPO는 아래의 loss를 최소화합니다:
        
        $$
        L_{GRPO}=−log⁡(\frac{e^R(y_{chosen})}{e^R(y_{chosen})+e^R(y_{rejected})})+λ⋅KL(π_θ∣∣π_{ref})
        $$
        
        - R(y): reward model이 생성한 chosen/rejected 응답에 대한 점수
        - πθ: 학습 중인 모델의 응답 분포
        - πref: 기준(reference) 모델의 분포
        - λ: KL regularization 계수
        
        **🧠 GRPO의 장점**
        
        | 항목 | 설명 |
        | --- | --- |
        | 🎯 더 명확한 학습 신호 | DPO는 "어떤 응답이 더 낫다"는 정보만 사용하지만, GRPO는 **점수 차이를 수치로 반영** |
        | 🤖 Reward model 재사용 | 새로운 domain/task에 맞는 reward model만 바꿔 재사용 가능 |
        | 🧩 멀티모달 적합 | image + text에 대한 응답 품질을 **통합적으로 평가 가능** |
        | 🧠 LLM 기반 reward model | 응답의 fluency, relevance, correctness 등을 LLM이 종합적으로 판단 |
        
        **🗣️ 실전 면접/인터뷰 답변 예시**
        
        > DeepSeek-VL R1에서 DPO 대신 **GRPO(Generative Reward-Preferring Optimization)**를 사용한 점이 인상 깊었습니다.
        > 
        > 
        > 이 방식은 단순히 chosen과 rejected 응답 간 preference만 학습하는 DPO의 한계를 넘어서, **explicit한 reward model이 각 응답의 점수를 평가하고, 그 점수 차이를 학습 신호로 활용**하는 구조입니다.
        > 
        > 특히 reward model이 generative LLM 구조를 따르기 때문에, 멀티모달 입력과 응답의 관련성, 설명력, 논리성 등을 **통합적으로 평가 가능**하다는 점에서 instruction-following fine-tuning에 매우 효과적이라고 느꼈습니다.
        > 
        > 향후 제 프로젝트에서도 cross-modal alignment나 image grounding 응답 평가에서, GRPO 방식의 reward model을 활용해 more controllable feedback 기반의 학습을 도입할 수 있을 것 같다는 인사이트를 얻었습니다.
        > 
        - 강화학습의 비용을 절감하기 위해 도입된 기법
        - 일반적으로 정책 모델과 동일한 크기의 크리틱 모델을 사용하는 대신, 그룹 점수를 통해 기준(baseline)을 추정
- **Q3.1.1 GRPO에서의 reward 모델은 뭘 사용했어?**
    
    GRPO에서의 reward model은 학습 대상 LLM과는 별개의 구조로, 일반적으로는 pretrained된 multimodal LLM LLava 스타일을 기반으로 응답의 적절성이나 reasoning 품질을 평가할 수 있도록 학습된 모델입니다.
    
    이 reward model은 학습 중에는 고정된 채로 사용되며, chosen/rejected 응답 쌍에 대해 score를 산출하고, 이 score 차이를 학습 신호로 활용하여 LLM이 더 나은 응답을 생성하도록 유도합니다. 이렇게 분리된 구조를 통해 **reward 기준의 일관성, 안정성, 설명력**을 확보할 수 있고, 다양한 domain에 대해 reward model만 바꿔서 재활용 가능한 점도 장점이라고 생각합니다.
    
    <aside>
    💡
    
    **1. RL과 GRPO**
    
    - **1.1. 기존 Supervised Fine-Tuning (SFT) 방식**
        - 전통적으로 **Supervised Fine-Tuning (SFT)**은 **사람이 레이블링한 데이터**를 기반으로 모델을 학습하는 방식
        - 즉, 사람이 제공한 정답이 있는 데이터셋을 사용하여 모델을 업데이트
        - 하지만 추론(reasoning) 능력을 강화하는 데 있어 SFT만으로는 한계가 있다.
    - **1.2. RL(Reinforcement Learning) 설명**
        - RL을 활용하여 모델이 스스로 논리적 사고를 발전시킬 수 있도록 유도
        - DeepSeek에서는 GRPO (Group Relative Policy Optimization)라는 RL 기법 사용
        - 사람이 레이블한 데이터가 필요 없음(모델이 스스로 데이터를 생성하고 비교하면서 학습)
        - **모델이 스스로 생성한 답변들끼리 비교하여 상대적인 성능 차이를 평가**하는 방식
        - 즉, 강화학습이란 모델 학습 방식이 하는 것임
            
            **스스로 연습하고 평가하고 진화**
            
    - **1.3. GRPO(Group Relative Policy Optimization) 특징**
        - Critic 모델을 제거하여 계산 비용 절감
            - Critic 모델을 제거하여 계산 비용 절감 (기존 기법에서는 critic model을 사용함, Critic model은 보상을 담당하는 모델)
            - 그룹 내 샘플 점수(**group scores**)를 활용해 성능을 평가함 (메모리 및 연산 비용이 낮아 효율적)
        - Group-based Advantage Estimation
            - RL에서 모델을 업데이트할 때, 특정 행동이 얼마나 유용했는지를 평가하는 것이 중요합니다. 이를 **Advantage**라고 함
            - 동일한 질문에 대해 여러 개의 답변을 생성하고, 이들 간의 상대적인 점수를 비교하는 방식 사용
            - 기존 RL에서는 “이 답변이 맞았나? 틀렸나?“를 평가하는 **절대적인 보상(absolute reward)** 방식을 사용. 그러나 DeepSeek에서는 절대적인 보상 값(reward)을 사용하는 것이 아니라 **그룹 내 상대적인 우위를 기반**으로 모델을 업데이트. 이 방식으로 점점 더 좋은 답변을 생성하도록 학습
        - KL-Divergence Penalty를 활용한 안정적인 학습
            - 새로운 정책(policy)이 급격하게 변화하는 것을 방지하기 위해 **KL-Divergence Penalty(KL 발산 패널티)**를 적용
    - **1.4. 수식**
        
        작동 방식
        
        - 각 질문 $q$에 대해, 기존 정책  $\pi_{\theta_{old}}$에서 그룹 출력  o1,o2,...,oG를 샘플링한다.
        - 정책 모델  $\pi_{\theta}$는 아래 목적함수  $J_{GRPO}(θ)$를 최대화 하는 방향으로 최적화 된다.
        - 목적함수는 정책 비율을  $1−ϵ$과  $1−ϵ$사이로 제한하는 클리핑 기법을 포함하며, KL 발산 패널티 $D_{KL}$를 통해 정책 안정성을 유지한다.
        
        **목적함수**
        
        $J_{GRPO}(θ)=\mathbb{E}[\frac{1}{G}\Sigma_{i=1}^Gmin(\frac{π_θ(o_i∣q)}{π_{θ_{old}}(o_i∣q)}A_i,clip(\frac{π_θ(o_i∣q)}{π_{θ_{old}}(o_i∣q)},1−ϵ,1+ϵ)A_i)]−βD_{KL}(π_θ‖π_{ref})$
        
        용어 정리
        
        - E : 기댓값.
        - ∑i=1 부터 G 까지 합산.
        - min(...) : 최소값 계산.
        - clip(x,a,b): x를 a와 b사이로 제한.
        - $π_θ(o_i∣q)$ : 정책 함수 πθ, 새로운 정책이 주어진 질문 q 에서 출력 o_i 를 선택할 확률
        - πθold(oi∣q) : 이전 정책에서 주어진 질문q 에서 출력 o_i 를 선택할 확률
        - D_KL: KL 발산 패널티
        
        수식해석
        
        - 기댓값을 계산한 것에 패널티를 빼줌
        - 기댓값: 최소값의 평균
        - min(⋅,clip(⋅)): 클리핑 기법 - 학습 과정에서 이 비율이 너무 크거나 작아지는 것을 방지
        - $\frac{π_θ(o_i∣q)}{π_{θ_{old}}(o_i∣q)}$ ,정책 비율 (Policy Ratio) : 새로운 정책과 기존 정책 간의 비율
        
        **KL 발산 패널티**
        
        $$
        D_{KL}(π_θ‖π_{ref})=\Sigma_{o_i}(π_{ref}(o_i∣q)log⁡\frac{{π_θ}(o_i∣q)}{π_{ref}(o_i∣q)}−1)
        $$
        
        - ∑oi: oi에 대한 합산.
        - πref(oi∣q): 참조 정책 πref의 확률.
        - πθ(oi∣q): 현재 정책 πθ의 확률.
        - $\frac{π_θ(o_i∣q)}{π_{ref}(o_i∣q)}$: 두 확률의 비율.
        
        수식해석
        
        - Log의 분자와 분모차이가 클수록 값이 커지고, 비슷하면 값이 작아짐
            - log(10000/1) = 4
            - log(1/1) = 0
        - 기존정책 x (차이가 클수록 큰값, 비슷하면 작은값)
        - 위 내용을 모두 더함
        
        **이득(Advantage)**
        
        Ai=ri−mean({r1,r2,…,rG})std({r1,r2,…,rG})
        
        - ri: 개별 보상 값.
        - {r1,r2,…,rG}: 보상 값들의 집합.
        
        **2.2.2. Reward Modeling (보상 모델링)**
        
        Reward(보상)은 강화학습에서 최정화 방향을 결정함.
        
        rule-based reward system (규칙 기반 보상 시스템) 사용함
        
        주요 보상 유형은 아래와 같음
        
        **정확도 보상 (Accuracy Rewards)**
        
        - 모델의 응답이 정확한지를 평가
        - 수학 문제와 같이 결정적인 결과를 요구하는 경우, 모델은 특정 형식(예: 상자 안에 최종 답변)을 따라야 하며, 이를 통해 규칙 기반으로 정확성을 검증
        - LeetCode 문제의 경우, 컴파일러를 사용하여 사전에 정의된 테스트 케이스를 기반으로 피드백을 생성
        
        **형식 보상 (Format Rewards)**
        
        - 모델이 사고 과정을 "<think>"와 "</think>" 태그 사이에 작성하도록 강제하는 보상 모델을 활용
        
        보상에서 neural reward model(신경 보상 모델)은 사용하지 않음.
        
        보상 해킹(reward hacking)에 취약할 수 있으며, 보상 모델을 재학습하려면 추가 자원이 필요하고 훈련 파이프라인이 복잡해짐

- **Q4.2. LangChain과 llama-index의 역할 분리는 어떻게 구성하셨나요?**
    
    “저는 LangChain과 LlamaIndex를 각각의 장점에 따라 역할을 분리해서 썼습니다.
    
    `1` LangChain은 주로 전체 임베딩, workflow orchestration에 사용했어요. 예를 들어 **embedding 생성 후 검색, 검색된 결과를 prompt에 넣고 LLM으로 응답 생성**하는 RetrievalQA나 custom multimodal chain을 LangChain으로 조립했고, input/output 흐름을 제어하는 데 적합했습니다.
    
    `2` 반면 LlamaIndex는 **데이터 indexing과 문서 처리 abstraction**에 썼습니다. 예를 들어 **이미지나 가전 로그 기반 문서에 metadata를 붙여서 vector DB에 저장**할 때, 각 modality별 chunking과 schema 정의를 LlamaIndex에서 처리하고, vectorstore에 push하는 역할을 했어요.
    
    이런 식으로 분리하니까 실험 반복도 빠르고, 나중에 Web demo 구성하거나 다른 LLM으로 바꿔 실험할 때도 유연하게 대응할 수 있었습니다.”
